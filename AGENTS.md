# Repository Guidelines

このリポジトリは Acecore 公式サイトの Astro 静的サイトです。AI エージェントや自動化ツールが作業する場合は、このファイルを最初に確認してください。

## コミュニケーション

- GitHub issue、pull request、PR 本文、レビュー返信、作業報告など、ユーザーやチームが読む GitHub 上の文章は日本語で書く。
- ユーザーが明示的に別言語を求めた場合だけ、その言語を使う。
- PR タイトルは日本語で、変更の目的が一目で分かる表現にする。
- 失敗した検証、未実施の確認、外部要因による制約は隠さず PR 本文と作業報告に書く。

## Issue 対応

- Issue URL だけを渡された場合でも、issue 本文、コメント、チェックリストを確認して受け入れ条件として扱う。
- 実装前に既存の構成、同種コンポーネント、翻訳データ、CMS 設定を確認し、既存パターンに合わせる。
- 要件が曖昧でも、リポジトリ内の文脈から安全に判断できる場合は作業を進める。判断が危険な場合だけ質問する。
- Issue 対応 PR では、本文に `Closes #番号` または該当 issue への参照を含める。

## 実装方針

- Astro、TypeScript、UnoCSS、既存コンポーネントの構成を尊重し、差分は目的に必要な範囲に絞る。
- 関連のない整形、リファクタリング、生成物更新を混ぜない。
- 既存の未コミット変更や別 branch の作業を戻さない。
- UI 変更では、デスクトップとモバイルでテキストがはみ出さないこと、クリック可能要素が分かりやすいこと、既存デザインと馴染むことを確認する。
- 画像、外部リンク、計測属性、構造化データ、検索インデックスに影響する変更は、周辺機能への影響も確認する。

## CMS と多言語

- 日本語の固定ページ文言は `src/i18n/source/ja/**/*.json` を正とする。
- 翻訳ファイルは `src/i18n/translations/{locale}.json` に集約されている。日本語 source のキー構造と翻訳側のキー構造を揃える。
- サイト文言を CMS から編集可能にする場合は `public/admin/config.yml` も更新し、対応する JSON key とフィールド名を揃える。
- CMS の日時、キャンペーン、告知、募集枠など期間制御が必要な情報は、表示開始と表示終了を CMS から扱える設計にする。
- 翻訳作業だけを依頼された場合は、日本語 source を勝手に変更しない。placeholder、URL、コード風 token、製品名は壊さない。
- ブログやニュースを増やす場合は、タグ、関連記事、内部リンク、CTA、Pagefind への影響を確認する。

## 検証

- サイト出力に影響する変更では原則 `npm run build` を実行する。
- Markdown、JSON、YAML、Astro、TypeScript を変更した場合は、対象ファイルに対して Prettier check を行う。既存の unrelated な整形差分を広げない。
- コミット前に `git diff --check` を実行する。
- UI 変更では可能な限りブラウザで表示確認する。ブラウザや Playwright が使えない場合は、その理由と代替確認を PR に記載する。
- ドキュメント、GitHub テンプレート、エージェント指示だけの変更では、`npm run build` は不要。代わりに対象ファイルの format check と `git diff --check` を行う。

## PR 作成

- PR は draft で作成してよい。ユーザーが ready を求めた場合だけ ready for review にする。
- PR 本文は `.github/pull_request_template.md` に沿って日本語で書く。
- PR 本文には、概要、主な変更点、確認したこと、未確認事項または補足を含める。
- 実行したコマンドは省略せず書く。実行していない検証は「未実施」と明記する。
- 変更が issue に対応する場合は、PR 本文に issue 番号を含める。
