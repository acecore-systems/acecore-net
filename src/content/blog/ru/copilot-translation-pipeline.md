Если говорить прямо, на этом сайте вы можете опубликовать одну японскую статью через Pages CMS и последовательно подготовить статьи блога на японском + 8 других языках. GitHub Actions и GitHub Copilot берут на себя создание задач перевода, создание PR переводов, сборку, автоматическое слияние и закрытие родительских задач.

В ежедневной работе нужно работать только с японскими статьями и информацией об авторах. Поскольку вам больше не нужно каждый раз вручную создавать задачи перевода или организовывать PR, нагрузка на управление многоязычным блогом значительно снижается.

## Предварительные условия для этого подхода

Этот подход предполагает, что у вас уже есть следующая инфраструктура на стороне Astro.

- Маршрутизация на 9 языках (ja, en, zh-cn, es, pt, fr, ko, de, ru)
- Запасной вариант для отображения японского на страницах без перевода
- Операции по обновлению японских статей и информации об авторах через Pages CMS

Как создать инфраструктуру описано в [Поддержка 9 языков в сайте Astro 6 — Автоматический перевод 136 статей блога и многоязычная архитектура](/blog/astro-i18n-blog-translation/). Эта статья сосредоточена только на том, как наложить автоматизированные операции перевода Copilot поверх этого.

## Что можно делать

Со стороны операций обычно есть два экрана, с которыми вы взаимодействуете. На этот раз мы используем экран Pages CMS как есть, чтобы сразу было понятно, **где взаимодействовать в ежедневных операциях**.

![Экран списка японского блога Pages CMS](/uploads/pagescms-blog-ja-live-20260329.png)

Первый экран — это список японского блога Pages CMS. Здесь вы просматриваете даты публикаций и авторов, добавляя и обновляя только японские статьи. Ключ в том, чтобы направить операции на **прикосновение только к японскому источнику перевода**, не заходя на экран редактирования нескольких языков каждый раз.

![Экран формы информации об авторе Pages CMS](/uploads/pagescms-authors-live-20260329.png)

Второй экран — это форма информации об авторе. Обновляя только поля на основе японского в CMS для данных об авторах и оставляя перевод `i18n` автоматизированному потоку GitHub, разделение операционных обязанностей становится довольно чистым.

## Когда этот подход работает лучше всего

Этот подход особенно эффективен для команд или сайтов следующих типов.

- Хотят использовать японский в качестве источника перевода
- Блог управляется в Markdown
- Вручную создавать задачи перевода каждый раз обременительно
- Готовы до определённой степени доверить ИИ качество перевода
- Но хотят останавливать PR, которые не проходят сборку или остаются черновиками

С другой стороны, если каждый язык имеет полностью независимую структуру редактирования, более подходящим может оказаться другой подход.

## Шаг 1. Зафиксировать источник перевода на японских статьях

Первое, что нужно решить, это "какой файл использовать в качестве источника перевода." Если это неясно, автоматизация рушится.

"Источник перевода" в этой статье означает **японский файл, который редактируется первым и служит стандартом для статей и производных данных на каждом языке**.

В данной конфигурации следующие элементы разделены на источник перевода и цель перевода.

- Источник перевода статей блога: `src/content/blog/{slug}.md`
- Цель перевода статей блога: `src/content/blog/{locale}/{slug}.md`
- Источник перевода информации об авторах: `src/data/authors.json`
- Цель перевода информации об авторах: `i18n` в `src/data/authors.json`

Структуру каталогов удобнее примерно в таком виде.

```text
src/content/blog/
  my-post.md
  another-post.md
  en/
    my-post.md
  zh-cn/
    my-post.md
  fr/
    my-post.md
```

Ключ в том, чтобы **выровнять slug файлов перевода с оригинальной японской статьёй**. Только это облегчает автоматическую идентификацию цели перевода из пути к источнику.

В этом репозитории, даже если файлы перевода ещё не существуют, URL-адреса языков сами генерируются с японским в качестве запасного варианта. Это означает, что вы можете работать в режиме "сначала опубликовать японскую статью, затем дать PR переводов подтянуться."

## Шаг 2. Преобразование push японских статей в задачи перевода

Следующий шаг — обнаружение изменений в японских статьях с помощью GitHub Actions и автоматическое создание задач перевода.

Минимально вам нужно следующее.

- Отслеживать push в `main`
- Нацеливаться только на `src/content/blog/*.md` и `src/data/authors.json`
- Обновлять существующую открытую задачу с тем же путём к источнику, а не создавать новую
- Встроить путь к источнику как маркер в тело задачи

Например, вставка такого комментария в тело задачи позволяет повторно использовать его в последующей автоматизации.

```md
<!-- translation-source:src/content/blog/my-post.md -->
<!-- translation-kind:blog-post -->
```

Базовая фильтрация на стороне рабочего процесса выглядит так.

```yaml
on:
  push:
    branches:
      - main
    paths:
      - src/content/blog/*.md
      - src/data/authors.json
```

Важно здесь не "создавать переводы напрямую", а **сначала создать задачу**. Вставляя шаг задачи, вы можете зафиксировать путь к источнику, целевые языки и условия перевода в форме, видимой как людям, так и ИИ.

## Шаг 3. Автоматически назначать задачи перевода Copilot

Создание только задачи всё ещё оставляет ручную работу, поэтому здесь вы автоматически назначаете Copilot.

Нужно сделать две вещи.

1. Добавить `COPILOT_AGENT_TOKEN` в секреты репозитория
2. Вызвать API назначения после создания задачи

Концептуально вы патчите задачу и устанавливаете Copilot в качестве исполнителя.

```json
{
  "assignees": ["copilot-swe-agent[bot]"],
  "agent_assignment": {
    "target_repo": "OWNER/REPO",
    "base_branch": "main",
    "custom_instructions": "Translate the Japanese source article..."
  }
}
```

На этом этапе хранение `custom_instructions` отдельно для статей и информации об авторах стабилизирует точность. Указание, что информация об авторах должна касаться только `i18n` в `src/data/authors.json`, а статьи должны создавать файл с тем же именем в `src/content/blog/{locale}/`, снижает количество ошибок.

## Шаг 4. Сборка PR переводов и автоматическое слияние

Эту часть безопаснее не делать безусловной автоматизацией. Рекомендация — объединять только PR, удовлетворяющие всем следующим условиям.

- PR создан Copilot
- Заголовок начинается с `[translation]`
- Нацелен на `main`
- Не черновик
- Сборка прошла успешно

В данной конфигурации это разделено на два этапа.

1. `Translation PR Build`
2. `Merge Translation PR`

Когда PR становится готов к проверке, строится его head, и в случае успеха выполняется squash merge. Поскольку это не зависит от защиты веток GitHub, удобно работать даже с небольшими репозиториями.

### Условия для ограничения автоматического слияния

При добавлении автоматического слияния следующие условия являются минимально рекомендуемыми.

- Исключить всё кроме PR переводов
- Остановить, если сборка не прошла
- Остановить, пока это черновик
- Исключить PR, созданные кем-либо кроме Copilot

С этими четырьмя условиями можно в значительной мере избежать аварий, когда обычные PR разработки тоже оказываются вовлечены.

## Шаг 5. Автоматически закрывать родительские задачи перевода после слияния

Последнее, что делает операции чистыми, — автоматическое закрытие родительских задач.

Метод простой — для слитых PR переводов выполните следующее.

1. Получить изменённые файлы в PR
2. Прочитать путь к источнику в теле PR
3. Найти открытые задачи, соответствующие маркеру `translation-source:`
4. Добавить комментарий и закрыть

Причина также смотреть путь к источнику в теле PR состоит в том, что в зависимости от ситуации обратный поиск источника может быть слабым, если смотреть только на изменённые файлы PR, созданных Copilot. **Использование как изменённых файлов, так и тела PR** даёт стабильные результаты.

## Дополнительные замечания

### Направление текста PR и задач Copilot на японский

Если вы хотите стабилизировать язык вывода Copilot на стороне GitHub, использование инструкций на уровне репозитория — самый прямой подход.

Просто разместите `.github/copilot-instructions.md`.

```md
This repository is an Astro static site for Acecore, deployed on Cloudflare Pages.

- For GitHub issues, pull requests, issue comments, pull request descriptions, review summaries, and other user-facing GitHub text, write in Japanese by default unless the task explicitly requires another language.
- For multilingual content work, treat Japanese source files as canonical and keep translated frontmatter aligned with the Japanese source.
```

Только этот один файл значительно стабилизирует язык по умолчанию и контекст, когда агент кодирования Copilot создаёт задачи и PR.

## Итог

Ключ этой конфигурации — превратить перевод из "того, о чём люди просят каждый раз" в **рутинный процесс, зависящий от push японского источника**.

Вот процесс ещё раз.

1. Написать только японскую статью
2. Push автоматически создаёт задачу перевода
3. Автоматически назначить Copilot
4. Собрать PR перевода и автоматически слить
5. Автоматически закрыть родительскую задачу тоже

Как только это настроено, операционное ощущение довольно прямолинейное. **Просто опубликуйте японскую статью, и статьи на других языках будут последовательно завершаться на стороне GitHub**.

Конечно, на практике асинхронный процесс создания задачи, выполнения Copilot, создания PR, сборки и слияния занимает время, поэтому всё не происходит "мгновенно." Однако операционному персоналу больше не нужно каждый раз вручную создавать задачи перевода или забывать закрывать PR.

Эта статья сама по себе структурирована так, чтобы её можно было подать в этот процесс с японской версией в качестве основы. Если вы непрерывно управляете многоязычным сайтом, начать с этого уровня автоматизации — наверное, самый подходящий вариант.
